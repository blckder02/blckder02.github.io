URLDNS

这条链准确的说达不到”利用”的目的，只是适合在检测反序列化漏洞时使用；
通过初始化⼀个java.net.URL对象，作为 key 放在java.util.HashMap中；然后，设置这个 URL 对象的 hashCode 为初始值 -1 ，这样反序列化时将会重新调用 hashCode()方法计算它的 hashCode 值，在该方法调用的过程中，就会触发DNS请求。（getByName(host)触发）

CC链

CC 1-7的链涉及两部分 Commons Collections 版本；

common collections3.x (3.1-3.2.1)
版本有CC1、3、5、6、7；

cc1有两条利用链，lazymap链和transformedmap链，核心触发点都是InvokerTransformer 类里面的 transform 方法可以通过反射调用任意类的任意方法来实现RCE；
它们的不同点在于一个是当调用lazymap#get()方法找不到值时调用transform()，一个是当使用transformedmap#put()写入元素时调用transform()；
cc1这两条链只能在 JDK 版本小于 8u71时才能用，因为 8u71 之后对两条链都用到的AnnotationInvocationHandler类的 readObject 方法做了更改。（无法再执行 set 或 put 操作）

cc3的触发过程是通过LazyMap.get()方法调用到transform()方法，然后使用 javassist 生成一个恶意类，把恶意类的字节码存入了TemplatesImpl对象中，调用了 transform()方法后会实例化这个对象，同时就加载了恶意字节码，从而触发RCE；
前半部分调用到 transform() 方法的过程和cc1一样可以使用lazymap或者transformedmp链，所以仍然只能在 JDK 版本小于8u71时使用。

cc5则是绕过了 JDK 版本的限制，从 BadAttributeValueExpException 的 readObject() 方法入手来调用到 lazymap#get() ，从而调用到 transform() 方法。（TiedMapEntry.toString() 到 lazyMap）

cc6是从 HashMap 的 readObject() 方法入手，通过TiedMapEntry.getValue() 调用到 lazymap#get()。

cc7 是从Hashtable.readObject() 方法入手，通过AbstractMap.equals() 方法调用到 lazymap#get()。

commoncollections4.x (4.0-4.3) CC2、CC4
4.0版本中对LazyMap类做了修改（删除了lazyMap#decode()）

cc2是从 PriorityQueue.readObject() 入手，通过TransformingComparator.compare() 方法调用到 transform() ，然后使用 javassist 生成恶意类，将其存入了TemplatesImpl对象中，最后被实例化加载触发RCE。
cc4相当于CC2与CC3的结合（和CC2一样从PriorityQueue.readObject() 入手，通过TrAXFilter#TrAXFilter() 来实例化 javassist 生成的恶意类。）

Fastjson

这系列漏洞产生的主要是跟fastjson反序列化的特点有关，可以在 json 数据中通过@type字段来指定反序列化返回的类，一旦有参数可控、可利用的类，就可以传入恶意参数并且反序列化为指定类，从而造成反序列化攻击。
Fastjson的防御措施是从1.2.25版本开始增加了一个autotype字段来开关自动检测，默认是开启的，还有checkAutoType()方法来对反序列化的类型进行黑白名单校验。
攻击者对它的绕过方式也主要是围绕黑白名单展开。

1.2.24版本
这个时候还没有添加 autotyte，可以指定反序列化类型为 TemplatesImpl，这个类的一个成员变量的 get() 方法中对_bytecodes参数进行了实例化，这个参数是可控的，所以将一个恶意类的字节码 base64 编码后传进去，反序列化时会调用成员变量的 get() 方法，从而加载并实例化恶意类。

<=1.2.41
从1.2.25版本开始增加了类型检测，将前面使用的类加入了黑名单。
但是对类名只做了一次简单检测，通过对类名添加指定特殊字符（比如在类的前面加L，后面加;），即可绕过，在加载过中会自动把字符去除，不影响正常反序列化。

1.2.42
这个版本是将类名中的L和;去除后再做的黑名单校验。
但是只去除了一次，双写即可绕过。

1.2.43
它对类名中的L和;进行了重复去除，再进行黑名单校验。
但是还有一种字符绕过方式，就是在类名前面添加[，后面添加[{，在加载过程中这些字符也会被自动去除，不影响反序列化。（这是根据代码逻辑推出来的）

<=1.2.45
修复了两种字符绕过的方式。寻找黑名单中没有的类进行利用。

<=1.2.47
将之前利用的类都添加进了黑名单。
这里有一种通用的绕过方式，当反序列化的对象是java.lang.Class时，会调用TypeUtils.loadClass()，它有一个cache值默认为 true，然后就会把Class类的val参数添加进一个mapping中，这里就可以把val参数设为之前可利用的类，无论是否在黑名单中。当调用 checkAutotype() 时，就能直接从mapping中获取到可利用的类，就绕过了黑名单校验。

<=1.2.68
把java.lang.Class也加入了黑名单，只能寻找其他黑名单中没有的可利用类。

<=1.2.80
(可以在特定条件下绕过 AutoType 关闭限制加载远程对象进行反序列化)

有好几条利用链，但是差不多都是一个原理；
构造两条 json 数据，第一条里写入两个@type字段，第一个值为java.lang.Exception，第二个type得是一个可利用的Exception类型，这个类需要有一个对象类型的成员变量，并且这个对象类会从指定远程地址中加载资源，这样就能在远程地址放一个恶意类，会被加载尽力，然后成员变量的 set 或 get 方法是中会实例化这个恶意类。

第二条 json 数据就是指定@type类型为前面的对象类，并且传入含有恶意类的远程地址。

也就是说要通过java.lang.Exception引入特殊的 Exception 类，从而将它的成员变量类型添加进 mapping，构造一个异常，捕获异常后就反序列化第二天 json 数据，直接从 mapping 中获取利用类，绕过黑名单检测，进而加载并实例化远程地址的恶意类。

Jackson

Jackson中是通过DefaultTyping字段来控制反序列化类型的自动检测，默认是关闭的，需要显示的开启才能按照指定的类型进行反序列化，不像fastjson中的 Autoype 默认开启，更加安全。

Spring

Spring Security
Spring Security 是一种包含身份验证、授权和访问控制的安全框架。
当它允许forward和include调度类型时，应用程序就可能通过转发或包含将请求跳转到更高的受保护的权限端点，从而达到权限绕过的目的。

比如页面a是允许匿名访问的页面，但是在这个页面上进行了转发操作，转发到了页面b，页面b是一个需要权限认证才能访问的页面，但是通过a页面来访问b页面就只会验证访问a页面的权限，不会验证访问b页面的权限。

Shiro

Shiro身份验证流程：

  1. 用户通过用户名和密码登录应用程序
  2. Shiro 会从应用程序的数据存储中检索用户的账号信息，以及与该账号相关的角色或权限
  3. Shrio 验证用户输入的密码与存储的密码是否一致
  4. 如果密码准确， Shiro就会创建一个 Sunbject 实例来表示经过身份验证的用户
  5. Subject实例与用户的角色和权限相关联
  6. 应用程序就可以通过Subject实例来对用户进行访问控制

为什么不想在原公司干了

当初选择安全研究岗位是因为在学校里面学习安全的时候接触到技术方面的知识比较多，也觉得从事安全研究行业能更深入的了解某些技术手段。但是实习之后发现公司需要研究的东西和自己想的不是一回事，我干的事大部分就是漏洞分析和漏洞挖掘方面，我觉得这两件事过于单一，没有直接的价值体现。而且乙方的安全大多都是为甲方安全服务的。现在有机会到甲方从事安全工作，当然是自身价值最直接的体现。
在甲方能够直观的了解整个安全体系的建设过程并且参与其中，能够清晰的知道每一项安全工作对公司的重要性。也有更多的机会的让我接触到企业安全的真实场景。
而且原神是我一直在玩的一款游戏，能够进入米哈游工作也算是游戏迷的一个愿望。
（想深入了解米哈游的文化理念，希望能成为其中的一份子。）

为什么选择运营岗

我有了解到安全运营岗位能接触到的安全环节比较多，从业务开发到风险修复，每个环节都能学到很多东西，对自身的能力提升有很大作用。而且我本来也是做漏洞方面的，对业务潜在的漏洞和已发现的漏洞都能充分理解，从而能很好地协调各部门配合对漏洞进行修复。算是将我的漏洞知识运用到实际的企业场景中，这也是我想达到的一个目标。
